## 3. 测试覆盖率 95% 怎么理解

## 3.1 先说现实结论

如果你问的是：

- `整个仓库所有类的全量覆盖率 > 95%`

那对当前这个项目来说，短期内并不现实，也没有必要为了这个数字去写大量无意义测试。

原因很现实：

1. 你这个项目已经有多个微服务、Mapper、Controller、Config、VO、DTO、MQ、缓存链路。
2. 很多类是配置类、数据对象类、框架接入类，本来就不适合用“高覆盖率”去衡量价值。
3. 如果为了全仓 95% 去覆盖所有 getter/setter、VO、启动类，只会制造测试噪音。

所以在真实项目里，更合理的理解是：

### 覆盖率 95% 应该优先要求在“核心链路”和“关键业务代码”上实现

比如：

- 登录鉴权链路
- token 失效校验链路
- 点赞收藏幂等逻辑
- 评论创建链路
- 通知生成链路
- 权限校验链路

## 3.2 当前项目更合理的覆盖率目标

建议分层看：

### 核心认证与会话链路

目标：

- `95%+`

包括：

- JWT 解析
- Redis token 校验
- Redis 失效降级
- logout 失效

### 核心业务服务

目标：

- `80%+`

包括：

- UserService
- ArticleService
- CommentService
- NotifyService

### Controller / DTO / VO / 启动类

目标：

- 不强求 95
- 有冒烟和集成验证即可

## 3.4 为什么我不建议你现在硬追全仓 95

面试官真正想看的是：

- 你有没有测试意识
- 你有没有分层测试设计
- 你能不能说清楚哪些代码值得重点测

而不是单纯报一个数字。

所以你在面试里可以这样说：

“我没有机械追求所有类 95% 覆盖率，而是优先把认证、评论通知、幂等控制这些核心链路做高覆盖测试；对配置类和简单数据对象则更多依赖集成测试与冒烟测试。”

这个回答比盲目说“全仓 95%”更像真实工程实践。

## 7. 面试高频问题：token 在 Redis 里，如果 Redis 挂了怎么办

这是很典型的面试题，而且你现在这个项目正好能讲。

## 7.1 先给结论

### 只把 token 真值强依赖 Redis，并不是最完美的设计

因为这样做会带来一个问题：

- Redis 挂了
- 所有受保护接口都可能一起受影响

所以如果是生产系统，必须考虑 Redis 异常时的降级策略。

## 7.2 当前项目原来有什么风险

你现在的认证链路原来是：

- 登录签 JWT
- token 缓存在 Redis
- 各微服务每次请求都去 Redis 校验 token 是否当前有效

优点：

- 支持退出登录立即失效
- 支持踢人下线
- 支持单点统一会话控制

缺点：

- Redis 一挂，认证中心就受影响

这在现实里是有风险的。

## 7.3 现在我已经做了什么改进

这次已经做了降级处理：

### 1. JWT 仍然是第一层凭证

- token 自身有签名和过期时间

### 2. Redis 是“会话增强校验”

- 用来做 logout 立即失效
- 用来做 kickout
- 用来做单点会话控制

### 3. Redis 失效时加入降级策略

当前 `TokenSessionValidator` 已经支持：

- Redis 校验成功时，把最近验证过的 token 记到本地短期缓存
- Redis 短时故障时，优先使用本地短期缓存继续放行
- 如果本地也没有，再根据配置决定是否 fail-open

关键文件：

- [TokenSessionValidator.java](/F:/test_file/blog-cloud/blog-common/src/main/java/com/blogcommon/auth/TokenSessionValidator.java)

当前配置：

```yml
auth:
  token-session:
    fail-open-on-redis-error: true
    local-cache-seconds: 300
```

这表示：

- Redis 短时异常时，系统优先保可用性
- 已经验证过的 token 可以在本地短时间继续通过

## 7.4 这样设计有没有问题

### 有权衡，但不是错误设计

现实中这属于：

`可用性优先的认证降级设计`

适合：

- 内容社区
- 博客论坛
- 用户操作不是特别高风险的系统

不适合：

- 银行
- 支付
- 风控
- 强安全后台

因为在这些高安全场景里，Redis 异常时更可能选择：

- 直接拒绝所有受保护请求
- 也就是 fail-close

## 7.5 现实项目里更成熟的改进方案

如果以后要做得更像线上系统，可以进一步升级成：

### 方案 1：短效 Access Token + 长效 Refresh Token

做法：

- Access Token 自包含，5~15 分钟有效
- Refresh Token 存 Redis 或数据库

好处：

- Redis 挂了时，短时间内 access token 仍可工作
- logout 的即时性略有损失，但系统可用性更高

### 方案 2：Redis + 本地缓存 + 多级降级

做法：

- Redis 作为主会话状态
- 本地缓存保存最近活跃 token
- Redis 挂了时短期使用本地缓存托底

这就是你当前项目现在采用的方向。

### 方案 3：不同接口采用不同策略

例如：

- 普通内容接口：fail-open
- 管理接口：fail-close

这样能兼顾安全和可用性。

## 7.6 面试里怎么答这题

你可以这样回答：

“如果 token 的有效性完全依赖 Redis，那么 Redis 挂掉时会影响所有受保护接口，所以这类设计不能只考虑 logout 的即时失效，还要考虑认证可用性。我现在这个项目里采用的是 JWT + Redis 会话增强校验的方式，平时通过 Redis 实现统一 logout 和单点会话控制；但为了避免 Redis 短时故障导致所有微服务不可用，我又加了本地短期 token 缓存和 fail-open 降级策略。对于博客论坛这类低风险业务，这样的权衡是合理的；如果是支付或后台高权限场景，我会改成更严格的 fail-close 或短效 access token + refresh token 方案。” 

这个回答是比较成熟的。

## 8. 当前项目的最终建议

接下来最建议你做的事：

1. 保留当前 Redis 降级方案，先满足项目上线和面试表达。
2. 用 [integration-smoke.ps1](/F:/test_file/blog-cloud/deploy/integration-smoke.ps1) 作为上线前冒烟脚本。
3. 用 GitHub Actions 跑单元测试和打包。
4. 下一阶段再把认证升级成：
   - Access Token + Refresh Token
   - 管理接口 fail-close
   - 普通接口 fail-open

这样项目就更像真实生产系统的渐进式演进。
