## 4. CD 流水线现在怎么工作

原来 CD 只是：

- 打包
- 上传构建产物

现在已经扩成 3 段：

### 第一段：GitHub Actions 打包

- `mvn -B -DskipTests package`

### 第二段：上传到 Linux

使用：

- `appleboy/scp-action`

上传内容：

- 5 个 jar
- `restart-services.sh`
- `post-deploy-check.sh`

### 第三段：SSH 到 Linux 执行发布

使用：

- `appleboy/ssh-action`

执行动作：

1. 备份旧 jar
2. 覆盖新 jar
3. 覆盖重启脚本
4. 重启 5 个服务
5. 等待服务启动
6. 执行发布后检查

文件：

- [cd.yml](/F:/test_file/blog-cloud/.github/workflows/cd.yml)
- [restart-services.sh](/F:/test_file/blog-cloud/deploy/restart-services.sh)
- [post-deploy-check.sh](/F:/test_file/blog-cloud/deploy/post-deploy-check.sh)

---

## 5. 发布后检查做了什么

`post-deploy-check.sh` 当前会检查：

1. 网关健康检查
2. 文章分页接口
3. 版块列表接口
4. 未登录访问通知接口时是否正确拦截

这是一个轻量发布后检查，不是完整回归测试。

它的作用是：

- 快速判断“服务有没有起来”
- 快速判断“网关和业务路由有没有坏”
- 快速判断“基本鉴权是不是还在工作”

---

## 6. GitHub Secrets 需要准备什么

这个 CD 工作流落地时，需要在仓库里配置：

- `DEPLOY_HOST`
- `DEPLOY_PORT`
- `DEPLOY_USER`
- `DEPLOY_SSH_KEY`

如果没有这些密钥，workflow 能看懂，但不能真正发布。

---

## 7. 这套设计的工程价值

这套改造的价值不是“看起来高级”，而是解决 3 个真实问题：

### 第一，测试不再只靠手点

核心业务分支已经有单元测试兜底。

### 第二，发布不再只是上传 jar

现在已经具备：

- 上传
- 备份
- 重启
- 发布后验证

的完整链路。

### 第三，质量门槛更清晰

当前质量策略是：

- 认证链路用 `JaCoCo` 做 `95%+` 硬门槛
- 核心业务 service 用单元测试覆盖主干和错误分支
- 系统级联调用自动化冒烟脚本做兜底

这比单纯追求“全仓覆盖率数字”更符合真实项目实践。
