# 部署文档与上线流程

## 1. 文档定位

这份文档只讲两件事：

1. 这个项目怎么部署到 Linux 服务器。
2. 一次正式上线应该按什么顺序执行。

---

## 2. 部署架构

当前项目推荐的基础部署结构：

- Nginx：对外统一入口，做反向代理和 HTTPS
- blog-gateway：网关层，统一鉴权和路由分发
- user-service：用户、登录、权限
- article-service：文章、点赞、收藏、热榜
- comment-service：评论、回复、评论消息投递
- notify-service：通知、未读数
- MySQL：主业务数据
- Redis：缓存、token 会话增强、热度统计
- RabbitMQ：评论通知异步解耦
- Nacos：服务注册发现

推荐访问路径：

```text
浏览器 / Postman
        |
      Nginx
        |
   blog-gateway
        |
  lb://service-name
        |
各业务微服务
```

---

## 3. 服务器准备

建议至少准备：

- 1 台 Linux 服务器
- JDK 17
- Maven 3.9+
- MySQL 8
- Docker / Docker Compose
- 可选：Nginx

如果是最小可运行方案：

- MySQL 可以单独装在宿主机
- Redis、RabbitMQ、Nacos 可以先继续放在 Docker 里
- 五个 Spring Boot 服务以 jar 方式运行

---

## 4. 打包步骤

在项目根目录执行：

```bash
mvn clean package -DskipTests
```

生成的 jar 在各模块的 `target` 目录：

- `user-service/target/user-service-1.0-SNAPSHOT.jar`
- `article-service/target/article-service-1.0-SNAPSHOT.jar`
- `comment-service/target/comment-service-1.0-SNAPSHOT.jar`
- `notify-service/target/notify-service-1.0-SNAPSHOT.jar`
- `blog-gateway/target/blog-gateway-1.0-SNAPSHOT.jar`

---

## 5. Linux 目录建议

建议部署目录如下：

```text
/opt/blog-cloud/
├─ apps/
│  ├─ user-service.jar
│  ├─ article-service.jar
│  ├─ comment-service.jar
│  ├─ notify-service.jar
│  └─ blog-gateway.jar
├─ logs/
├─ scripts/
└─ backup/
```

其中：

- `apps` 放当前运行包
- `logs` 放运行日志
- `scripts` 放启动、停止、重启脚本
- `backup` 放上一个稳定版本 jar，方便回滚

---

## 6. 上传方式

常见有三种：

1. `scp` 或 `sftp` 手工上传
2. Jenkins 自动上传
3. GitHub Actions 打包后经 SSH 发布

最简单的手工方式：

```bash
scp blog-gateway/target/blog-gateway-1.0-SNAPSHOT.jar root@server:/opt/blog-cloud/apps/blog-gateway.jar
```

其他 4 个服务同理。

---

## 7. 启动方式

### 7.1 中间件先启动

先保证这些组件已经启动：

- MySQL
- Redis
- RabbitMQ
- Nacos

### 7.2 服务启动顺序

建议顺序：

1. `user-service`
2. `article-service`
3. `comment-service`
4. `notify-service`
5. `blog-gateway`

### 7.3 启动命令

```bash
nohup java -jar /opt/blog-cloud/apps/user-service.jar --spring.profiles.active=prod > /opt/blog-cloud/logs/user-service.out 2>&1 &
nohup java -jar /opt/blog-cloud/apps/article-service.jar --spring.profiles.active=prod > /opt/blog-cloud/logs/article-service.out 2>&1 &
nohup java -jar /opt/blog-cloud/apps/comment-service.jar --spring.profiles.active=prod > /opt/blog-cloud/logs/comment-service.out 2>&1 &
nohup java -jar /opt/blog-cloud/apps/notify-service.jar --spring.profiles.active=prod > /opt/blog-cloud/logs/notify-service.out 2>&1 &
nohup java -jar /opt/blog-cloud/apps/blog-gateway.jar --spring.profiles.active=cloud,prod > /opt/blog-cloud/logs/blog-gateway.out 2>&1 &
```

## 8. Nginx 配置思路

对外不建议直接暴露网关进程端口，建议：

- 公网请求进 Nginx
- Nginx 转发到网关 `8080`

典型作用：

- HTTPS 证书
- 域名接入
- 基础限流
- 访问日志
- 统一跨域和代理入口

## 9. 正式上线流程

### 9.1 发布前

上线前要做 6 件事：

1. 确认发布分支和 commit
2. 跑单元测试
3. 跑覆盖率校验
4. 跑自动化冒烟脚本
5. 确认数据库脚本
6. 备份旧 jar 和关键配置

### 9.2 发布中

上线顺序建议是：

1. 上传新 jar
2. 执行数据库脚本
3. 重启服务
4. 检查 Nacos 注册
5. 检查网关是否可访问
6. 检查 Redis / RabbitMQ / MySQL 连接

### 9.3 发布后验证

发布后至少验证：

1. 登录
2. 发布文章
3. 查看文章详情
4. 评论
5. 通知未读数
6. 退出登录后旧 token 失效

这 6 步正好可以直接用：

- [integration-smoke.ps1](/F:/test_file/blog-cloud/deploy/integration-smoke.ps1)

## 10. 回滚流程

如果上线后发现严重问题，回滚顺序建议：

1. 停止当前服务
2. 恢复上一个稳定版本 jar
3. 恢复配置
4. 如果本次数据库脚本不可兼容，再按预案回滚数据库
5. 再次执行冒烟验证

回滚前必须提前准备：

- 上一版 jar
- 上一版配置
- 数据库脚本回滚方案

---

## 11. 当前项目适合的上线方式

按你现在这个项目规模，最合适的方式不是一上来就 Kubernetes，而是：

### 第一阶段

- Docker 起中间件
- Linux 跑 5 个 jar
- Nginx 转发到网关

### 第二阶段

- Jenkins / GitHub Actions 自动打包
- 自动上传服务器
- 执行重启脚本

### 第三阶段

- 再考虑 Docker 化微服务
- 再考虑容器编排和高可用

这个演进路线更符合当前项目体量，也更像真实小中型项目的落地过程。
