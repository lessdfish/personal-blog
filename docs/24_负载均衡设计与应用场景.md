## 1. 当前项目的负载均衡做到了哪一步

你这个项目现在的负载均衡不是停留在概念上，而是已经有两层真实链路：

### 第一层：网关层负载均衡

由 `blog-gateway` 负责。

网关路由已经不是写死端口，而是：

- `lb://user-service`
- `lb://article-service`
- `lb://comment-service`

## 2. 这次我补了什么

为了让这套负载均衡更像真实工程，我补了两类显式配置：

### 2.1 网关侧

在网关里补了：

- 负载均衡实例列表缓存
- 负载均衡重试开关
- 负载均衡日志级别

关键配置：

```yml
spring:
  cloud:
    loadbalancer:
      cache:
        enabled: true
        caffeine:
          spec: initialCapacity=64,maximumSize=512,expireAfterWrite=30s
      retry:
        enabled: true
```

### 2.2 comment-service 侧

在服务间调用层补了：

- Feign 调用超时控制
- 负载均衡实例缓存
- 失败重试开关

关键配置：

```yml
spring:
  cloud:
    loadbalancer:
      cache:
        enabled: true
        caffeine:
          spec: initialCapacity=32,maximumSize=256,expireAfterWrite=30s
      retry:
        enabled: true

feign:
  client:
    config:
      default:
        connectTimeout: 2000
        readTimeout: 3000
```

## 3. 它到底是怎么工作的

## 3.1 用户请求进入网关

例如前端访问：

```text
/api/article/page/normal
```

网关匹配到：

```text
lb://article-service
```

此时网关会去 `Nacos` 拿到 `article-service` 的实例列表，例如：

- `127.0.0.1:53101`
- `127.0.0.1:53102`
- `127.0.0.1:53103`

然后从里面选一个实例转发。

所以现在不是：

- 手工记端口

而是：

- 按服务名查实例
- 按实例列表做分发

## 3.2 服务间调用也是一样

例如 `comment-service` 要调用 `article-service` 更新评论数。

它不是写死：

```text
http://127.0.0.1:9002
```

而是用：

```java
@FeignClient(name = "article-service")
```

这样 Feign 会通过注册中心拿到多个 `article-service` 实例，再交给 `Spring Cloud LoadBalancer` 去挑选。

## 4. 负载均衡的作用

它的核心作用是 4 个。

### 第一，分摊请求压力

如果 `article-service` 启动 3 个实例，那么文章分页、文章详情、热榜查询这些请求就不会都压到一个进程上。

### 第二，提高整体吞吐

单实例处理能力有限，多实例后总吞吐通常更高。

例如：

- 单实例稳定 `100 QPS`
- 3 个实例一起工作，整体吞吐通常会更高

当然能不能线性提升，还要看：

- MySQL 会不会成为瓶颈
- Redis 和 MQ 会不会成为瓶颈

### 第三，提高可用性

如果一个实例挂了：

- 网关还能继续把流量转给其他实例
- 系统不会因为一个 Java 进程挂掉就整体不可用

### 第四，支撑弹性扩容

高峰期你可以多起几个实例，低峰期再收回。

这就是微服务真正有价值的地方之一：

- 按服务维度扩容
- 而不是整个单体一起扩

## 5. 在你这个项目里的真实使用场景

## 5.1 article-service

这是最典型需要做负载均衡的服务，因为：

- 文章列表
- 热榜分页
- 文章详情

这些都是高频读接口。

如果后面流量上来，最先扩的一般就是：

- `article-service`

## 5.2 comment-service

评论链路涉及：

- 写 MySQL
- 调文章服务更新评论数
- 发 MQ

高并发时评论服务压力也会明显上升，所以也很适合横向扩容。

## 5.3 user-service

用户服务平时不一定最重，但在这些时刻会明显升高：

- 大量用户同时登录
- 活动期间集中访问个人中心
- 管理端查询用户信息

所以它也适合单独扩容。

## 5.4 notify-service

如果后续你加：

- 消息中心
- 站内信
- WebSocket 推送

那么通知服务也会成为高频接口服务。

## 6. 实际开发里常见的多层负载均衡

真实项目里通常不是只有一层。

### 第一层：Nginx 负载均衡

面向公网入口：

```text
用户 -> Nginx -> Gateway(多实例)
```

作用：

- HTTPS
- 域名接入
- 公网入口限流
- 网关多实例高可用

### 第二层：Gateway 负载均衡

面向业务服务入口：

```text
Gateway -> article-service / user-service / comment-service / notify-service
```

作用：

- 把请求分发到不同业务服务实例

### 第三层：服务间调用负载均衡

面向微服务内部：

```text
comment-service -> article-service
comment-service -> user-service
```

作用：

- 内部调用也不打死单个实例

你现在这个项目已经具备第二层和第三层。

如果后面再加 Nginx 多网关实例，就会形成更完整的多层负载均衡结构。

## 7. 它不能解决什么问题

这点也要讲清楚。

负载均衡解决的是：

- 应用层实例分流
- 单实例故障转移

它不能直接解决：

- MySQL 慢查询
- Redis 热点 Key
- RabbitMQ 积压
- 大事务
- 大对象序列化开销

例如：

- 你把 `article-service` 扩成 3 个实例
- 但所有实例还是查同一个慢 SQL

那瓶颈仍然会卡在数据库。

所以真实项目里通常是：

1. 先做负载均衡
2. 再做数据库和缓存优化

## 8. 当前项目后续最合理的扩展方式

如果你要继续往“高并发 + 高可用”走，建议顺序是：

### 第一步

先把：

- `article-service`
- `comment-service`

做成多实例。

### 第二步

把 `blog-gateway` 也做成多实例，再由 Nginx 做入口层负载均衡。

### 第三步

如果数据库成为瓶颈，再继续做：

- 读写分离
- 热点缓存
- 分库分表

也就是说，负载均衡是应用层扩容的第一步，不是最后一步。

## 9. 面试里怎么回答

你可以这样说：

“我这个项目的负载均衡主要做了两层。第一层是在网关侧，网关通过 `lb://service-name` 配合 Nacos 注册发现，把外部请求分发到多个微服务实例；第二层是在服务间调用侧，comment-service 通过 Feign + Spring Cloud LoadBalancer 调用 article-service 和 user-service，也能按实例列表做分发。这样做的作用主要是分摊流量、提高吞吐、提升单实例故障下的可用性，并为后续按服务维度扩容预留基础。如果后续正式上线，我还会在公网入口再加一层 Nginx 负载均衡，形成入口层、网关层、服务层的多层负载均衡结构。”
