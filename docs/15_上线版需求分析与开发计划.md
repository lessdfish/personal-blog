# 上线版需求分析与开发计划

## 1. 文档目标

这份文档不是继续堆功能，而是先回答 4 个更重要的问题：

1. 当前项目已经做到哪一步。
2. 离“可以上线的系统”还差哪些东西。
3. 现在应该先做哪些基础功能，哪些功能放到上线后迭代。
4. 软件开发全流程里应该产出哪些文档，后续怎么按文档推进代码开发。

你的目标现在不是“做一个功能很多的论坛”，而是：

- 先做成一个能上线、能演示、能部署、能讲清楚的系统
- 再在第二阶段迭代板块、社区运营、推荐、搜索等增强功能

这才是更稳的路线。

## 3. 当前最核心的问题

你现在最大的风险不是“功能不够多”，而是“交付形态还不完整”。

也就是说，现阶段真正缺的是：

1. 没有完整前端
2. 没有完整上线版需求边界
3. 没有部署闭环
4. 没有统一的交付文档体系
5. 没有把当前系统收敛成一版稳定上线包

所以后续策略必须变成：

- 先冻结功能范围
- 先完成一版基础可上线系统
- 把增强功能全部放进二期迭代

## 4. 现阶段应该收敛成什么产品

## 4.1 产品定位

结合你当前技术栈和项目进度，最适合的产品定位不是“大型综合论坛”，而是：

`轻量级博客论坛系统 / 社区型内容平台`

它的最小上线目标应该是：

- 用户能注册登录
- 用户能发文章
- 用户能看文章
- 用户能评论
- 用户能收到通知
- 管理员能做基本管理
- 前端能完成主要页面闭环
- 系统能部署到 Linux 服务器运行

这就够形成一版完整项目。

## 4.2 一期上线必须有的功能

### 用户端

- 注册
- 登录
- 退出登录
- 修改个人资料
- 修改密码
- 查看个人信息

### 内容端

- 文章发布
- 文章详情
- 文章列表
- 热榜列表
- 点赞
- 收藏

### 互动端

- 评论文章
- 查看评论
- 删除自己的评论
- 查看通知
- 已读通知

### 管理端

- 用户基础管理
- 文章删除/下架
- 评论删除

### 系统端

- 网关统一入口
- JWT 鉴权
- Redis 缓存
- RabbitMQ 异步通知
- Nacos 注册发现
- MySQL 数据落库

## 4.3 一期暂缓的功能

这些功能不是没价值，而是现在不适合优先做：

- 复杂板块体系
- 举报审核流
- 搜索中心

- 推荐系统

- 多级审核
- 排行榜复杂算法

`二期迭代`

## 6. 结合当前项目，推荐的一期技术栈边界

你现在最适合的一期上线版技术栈，不要再无限扩。

推荐保留：

- `Spring Boot`
- `Spring Cloud Gateway`
- `Nacos`
- `MySQL`
- `Redis`
- `RabbitMQ`
- `Spring Security`
- `JWT`
- `MyBatis`
- `Docker Compose`
- `Nginx`

如果要补前端，推荐选一个：

- `Vue3 + Element Plus`

这样原因很简单：

- 对个人项目最稳
- 页面能快速搭出来
- 上手难度比自己从零搭管理后台低
- 和你的后端风格匹配

当前阶段不建议先上：

- Elasticsearch
- Kafka
- 分库分表
- Kubernetes
- 大型搜索/推荐系统

不是不能上，而是现在上会把项目复杂度拉高，影响你把“上线版闭环”做好。

## 10. 对当前需求做第 1 次深度分析：从“想用技术栈”转成“要解决什么问题”

你现在的原始诉求，其实不是产品需求，而是技术学习诉求：

- 想把技术栈都用到项目里
- 想学完整开发流程
- 想从创建项目到上线部署都经历一遍

这类诉求如果不转换，会出问题：

- 为了用技术而用技术
- 系统会越来越乱
- 最后项目像“技术拼盘”

所以第一次深度分析的结论是：

### 真正目标不是“技术栈越多越好”

而是：

- 用一套合理技术栈解决一套清晰业务问题
- 形成一版完整交付

### 当前最合适的项目目标

不是“做大而全论坛”，而是：

`做一个具备完整前后端、可部署、可上线的轻量社区博客系统`

---

## 11. 对当前需求做第 2 次深度分析：从“功能很多”转成“最小上线闭环”

你现在如果继续往系统里塞功能，会出现两个问题：

1. 上线时间会无限延后
2. 基础链路还没完全打磨稳定

所以第二次深度分析要做的是：

### 找出最小上线闭环

对你这个项目来说，最小闭环就是：

- 用户注册登录
- 文章列表/详情
- 发布文章
- 评论互动
- 通知查看
- 管理员基础管理
- 前端页面可操作
- Linux 可部署

只要这一套完整跑通，你的项目就已经具备非常强的展示价值。

### 结论

先不上：

- 分板块深度设计
- 复杂审核流
- 搜索中心
- 推荐系统

先把闭环做完。

---

## 12. 对当前需求做第 3 次深度分析：从“开发功能”转成“产品交付”

第三次深度分析最关键。

因为真正能上线的系统，不只是代码能跑，还包括：

- 有需求范围
- 有数据库初始化脚本
- 有前端页面
- 有部署方式
- 有配置管理
- 有测试报告
- 有上线方案

也就是说，你要从“我在写代码”，转成：

`我要交付一个系统`

### 这意味着后面开发顺序应该变

后续不应该再直接说“先加什么中间件”，而应该按下面顺序：

1. 定义上线版需求
2. 列页面清单
3. 列接口清单
4. 定数据库初始化脚本
5. 开发前端
6. 联调
7. 部署
8. 测试
9. 产出上线文档

### 第三次分析结论

你现在最需要的不是继续加功能，而是：

- 做产品收敛
- 做文档
- 做前端
- 做部署
- 做交付

## 14. 接下来应该怎么推进开发

现在最合理的顺序是：

1. 先基于本需求文档再拆一份“上线版功能清单”
2. 再出一份“前端页面清单 + 页面流转图”
3. 再出一份“接口清单 + 联调计划”
4. 然后正式进入前端开发和联调
5. 最后做部署文档和上线脚本

也就是说，下一步不应该直接继续往后端加功能，而应该：

`依赖需求文档进入具体的交付开发阶段`

## 15. 结论

当前项目已经不是“从 0 开始”，而是已经到了一个关键分叉点：

- 如果继续随意加功能，项目会越来越散
- 如果现在开始做需求收敛、前端补齐、部署闭环，你就会得到一个真正能上线的项目

所以从现在开始，后续开发应该围绕：

- 上线版需求
- 前端页面
- 联调
- 部署
- 文档交付

来推进，而不是再优先做复杂增强功能。
