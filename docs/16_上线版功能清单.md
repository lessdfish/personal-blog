# 上线版功能清单

## 1. 文档目标

这份文档用于把“上线版需求分析”进一步落成可执行的功能清单。

它解决 4 个问题：

1. 一期到底做哪些功能。
2. 哪些功能必须做，哪些功能可以延后。
3. 每个功能由前端还是后端承接什么内容。
4. 功能完成的验收标准是什么。

这份文档写完以后，后续开发不再按“想到什么做什么”，而是按功能清单推进。

---

## 2. 一期上线版本目标

一期版本目标不是做成“大而全论坛”，而是做成一个可以真实演示和部署的：

`轻量级社区博客系统`

要求达到：

- 游客可以浏览内容
- 用户可以注册登录和互动
- 作者可以发布文章
- 用户可以评论和接收通知
- 管理员可以做基础内容管理
- 系统可以部署到 Linux 服务器
- 前后端页面闭环完整

---

## 3. 功能优先级划分

为了避免后续开发失控，功能分成 3 类：

## P0：一期必须完成

这些功能不做完，就不能叫“可上线系统”。

## P1：一期可选增强

这些功能做了更完整，但不影响一期上线。

## P2：二期迭代功能

这些功能放到上线后再继续做。

## 4. P0 功能清单

## 4.1 用户与认证模块

### 功能 1：用户注册

功能说明：

- 用户输入用户名、密码、确认密码完成注册

后端职责：

- 校验用户名是否重复
- 校验密码格式
- 写入用户表

前端职责：

- 提供注册页面
- 做基础表单校验
- 注册成功后跳转登录页

验收标准：

- 合法用户可以注册成功
- 重复用户名注册失败
- 无效参数有明确提示

### 功能 2：用户登录

功能说明：

- 用户通过用户名密码登录，获取 token

后端职责：

- 校验用户名密码
- 签发 JWT
- 返回用户基础信息

前端职责：

- 登录页表单
- 登录成功后保存 token
- 跳转首页

验收标准：

- 正确账号密码能登录成功
- 错误账号密码有提示
- 登录后能访问需要登录的页面

### 功能 3：退出登录

功能说明：

- 用户主动退出当前登录状态

后端职责：

- 执行 token 刷新/失效相关逻辑

前端职责：

- 清空本地 token
- 跳转登录页

验收标准：

- 退出后再次访问受保护页面会被拦截

### 功能 4：个人信息查看与修改

功能说明：

- 查看当前登录用户信息
- 修改昵称、头像、简介等资料

后端职责：

- 查询当前用户资料
- 更新资料信息

前端职责：

- 个人中心页面
- 表单回显与提交

验收标准：

- 修改后重新进入页面能看到最新数据

### 功能 5：修改密码

功能说明：

- 用户修改自己的登录密码

后端职责：

- 校验旧密码
- 更新密码

前端职责：

- 修改密码页面

验收标准：

- 修改成功后旧密码失效，新密码可登录

## 4.6 系统基础能力

### 功能 24：网关统一入口

功能说明：

- 所有请求走统一网关

验收标准：

- 前端不直接调用各微服务端口

### 功能 25：JWT 鉴权

功能说明：

- 登录接口返回 token
- 受保护接口校验 token

验收标准：

- 未登录请求被拦截

### 功能 26：Redis 缓存

功能说明：

- 文章详情、热榜、计数等高频读场景使用 Redis

验收标准：

功能说明：

- 评论后异步生成通知

验收标准：

- 评论成功后能收到通知

功能说明：

- 服务按服务名注册
- 网关按服务名路由

验收标准：

- 服务能够出现在 Nacos 控制台中

功能说明：

- 提供接口文档查看入口

验收标准：

- 各服务 Swagger 页面可访问

## 9. 功能验收方式

每个功能上线前至少要经过 4 层验收：

1. 接口可用
2. 页面可操作
3. 权限正确
4. 数据正确

例如“发布文章”：

- 接口能成功写入文章
- 页面可正常提交
- 未登录不能发布
- 发布后文章能在列表看到

---

## 10. 与当前项目进度的匹配关系

当前项目后端已经覆盖了 P0 功能中的大部分接口能力。

当前最缺的是：

- 页面层
- 联调层
- 管理端页面
- 部署流程闭环

所以接下来的开发重点应该是：

1. 补前端
2. 做前后端联调
3. 做上线准备

而不是继续扩论坛玩法。

---

## 11. 当前开发建议

后续开发顺序建议固定为：

1. 先做“前端页面清单与页面流转图”
2. 再做“接口联调清单”
3. 再开始前端开发
4. 再做部署文档和上线脚本

这才是从需求走向交付的正确节奏。
