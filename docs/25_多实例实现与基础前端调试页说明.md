# 多实例实现与基础前端调试页说明

## 1. 多实例到底是怎么实现的

你问的这个问题本质上是：

### 是不是把相同的 jar 部署到不同服务器上

答案是：

### 是，也可以不是

多实例的本质不是“必须不同服务器”，而是：

- 同一个服务名
- 启动多个独立进程实例
- 这些实例都注册到注册中心
- 网关或其他服务按实例列表做分发

所以多实例有两种常见方式。

## 1.1 同机多实例

例如同一台机器上启动 3 个 `article-service`：

- `article-service-1`
- `article-service-2`
- `article-service-3`

它们运行的是：

- 同一份代码
- 通常也是同一个 jar

区别只是：

- 端口不同
- 进程不同

你这个项目现在已经支持这种模式，因为业务服务端口是：

```yml
server:
  port: ${SERVER_PORT:0}
```

也就是说：

- 不传端口就自动选空闲端口
- 多起几次同一个服务，就会注册成多个实例

## 1.2 多机多实例

更真实的线上方式是：

- 机器 A 跑一份 `article-service.jar`
- 机器 B 跑一份 `article-service.jar`
- 机器 C 再跑一份 `article-service.jar`

三台机器都注册到 `Nacos`

这时网关看到的仍然只是：

- `article-service` 有 3 个实例

它不关心这些实例是在同一台机器还是不同机器。

所以你可以直接这样理解：

### 多实例 = 多个相同服务进程

不一定非得多机器，但正式线上通常会部署到不同机器，避免单机故障把所有实例一起带走。

## 3. 调试页能做什么

## 3.1 User 页面

支持：

- 注册
- 登录
- 当前用户信息
- 用户上下文
- 退出登录
- token 校验

并且：

- 登录成功后会自动把 token 存到浏览器本地
- 后面其他页面会自动带上这个 token

## 3.2 Article 页面

支持：

- 普通分页
- 热榜分页
- 文章详情
- 发布帖子
- 点赞
- 取消点赞
- 收藏
- 取消收藏
- 查询点赞数
- 查询收藏数
- 查询浏览量
- 查询热度

## 3.3 Comment 页面

支持：

- 评论列表
- 评论树分页
- 创建评论
- 查询剩余评论次数

## 3.4 Notify 页面

支持：

- 未读数
- 通知分页
- 通知详情
- 单条已读
- 全部已读
- 删除通知

这些页面的目标不是做成真正前端，而是：

### 验证每个按钮点下去之后返回的 JSON 是否正确

## 5. 现在服务之间的调用是不是只有 comment 和 user

不是只有 `comment` 和 `user`。

当前已经存在的服务间调用是：

### comment-service -> article-service

用途：

- 创建评论前校验文章是否存在
- 判断文章是否允许评论
- 评论成功后回写文章评论数

### comment-service -> user-service

用途：

- 查询评论人和被回复人的简要信息

所以当前实际是：

- `comment-service -> article-service`
- `comment-service -> user-service`

---

## 6. 还有哪些地方适合互相调用

如果按你当前项目的发展方向，后续还适合增加这些调用。

## 6.1 article-service -> user-service

场景：

- 文章详情展示作者昵称、头像
- 文章列表展示作者摘要信息

现在文章侧很多地方还是只持有 `authorId`，如果以后前端要直接展示作者卡片，就很适合调用户服务拿作者简要信息。

## 6.2 notify-service -> user-service

场景：

- 通知列表展示发送人昵称和头像

现在通知详情里有 `senderId`，但如果前端想直接显示“谁评论了你”，就适合让通知服务调用用户服务补发信人信息。

## 6.3 article-service -> comment-service

场景：

- 文章详情页需要直接聚合评论摘要

但这个调用要谨慎，因为：

- 会增加文章详情接口的耦合
- 可能把详情接口拉慢

更合理的方式通常是：

- 前端分别调详情接口和评论接口

所以这个不一定要做成服务间调用。

## 6.4 user-service -> notify-service

场景：

- 个人中心首页需要显示未读通知数

但这类也不一定非得服务间调。

更常见做法是：

- 前端登录后自己单独调用通知未读数接口

这样耦合更低。

---

## 7. 我对你这个项目当前阶段的建议

当前阶段最适合保留的服务间调用是：

1. `comment-service -> article-service`
2. `comment-service -> user-service`

接下来如果要增强用户体验，优先再补：

3. `notify-service -> user-service`
4. `article-service -> user-service`

不建议现在就把所有服务互相打通，因为那样很容易把项目做成：

- 服务边界不清楚
- 调用链过长
- 排查问题更困难

对现在这个项目阶段来说，先保证：

- 核心链路短
- 服务边界清楚

更重要。
